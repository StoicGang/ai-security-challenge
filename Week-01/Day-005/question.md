#### You have two paragraphs and one user query. Why don't we directly compare the raw text? Why do we first convert both the paragraphs and the query into embeddings before performing semantic search?

>We convert both the documents and the user's query into embeddings because machine learning models compare numerical vector representations rather than raw text. These embeddings capture the semantic meaning of the text, allowing the system to retrieve relevant documents even when different words are used. Another advantage is that document embeddings can be generated once and reused for multiple searches, making semantic retrieval both meaningful and efficient.

#### Suppose you accidentally do this: Paragraphs are embedded using **Model A** and The user's query is embedded using **Model B**. You then compute cosine similarity. **Will the similarity scores still be meaningful? Why or why not?**

>No. Although cosine similarity can still compute a numerical value, that value is not meaningful because the paragraph and query embeddings were generated by different embedding models. Each model creates its own semantic space, so vectors from different models are not directly comparable. To perform reliable semantic search, both the documents and the queries must be embedded using the same embedding model.

#### What is the difference between Day 4 and Day 5?

>Day 4 focused on measuring semantic similarity between two known pieces of text. The similarity score itself was the final result. Day 5 extends that concept into semantic retrieval. Instead of comparing just two texts, we compare one query against multiple stored document embeddings, compute similarity scores for each, and retrieve the document with the highest similarity. In other words, semantic similarity becomes the mechanism for search.

#### **Imagine I give you 10 million documents.** A new user enters a search query. **Do we generate embeddings for all 10 million documents again every time someone searches?** Explain **why or why not**.

>No. The document embeddings are generated only once because the documents themselves do not change for every search. Those embeddings are stored and reused for future queries. When a user searches, only the user's query is converted into an embedding. That query embedding is then compared against the stored document embeddings to retrieve the most relevant documents. This avoids recomputing embeddings for millions of documents every time a search is performed.
