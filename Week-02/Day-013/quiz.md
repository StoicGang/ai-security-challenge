1. Without describing your code, explain what happens from the moment a user asks a question until the LLM generates the final answer. Focus on the flow of information through the RAG pipeline.
> A RAG pipeline has two phases: **indexing** and **retrieval**.
>During indexing, documents are split into chunks, each chunk is converted into an embedding using an embedding model, and those embeddings are stored in a vector database.
>When a user asks a question, the same embedding model converts the query into an embedding. The vector database performs a similarity search to retrieve the most relevant document chunks. Those retrieved chunks are combined with the user's question and system instructions to create a prompt. Finally, the LLM generates an answer using only the retrieved context, producing a grounded response.

2. What is the responsibility of the retriever, and what is the responsibility of the LLM? Why is separating these responsibilities important?
>The retriever's responsibility is to find the most relevant information from the knowledge base using semantic similarity. It retrieves the document chunks that best match the user's question.
>The LLM's responsibility is to read the retrieved context, understand it, and generate a natural-language answer. It does not search the knowledge base itself.
>Separating these responsibilities is important because each component specializes in a different task. The retriever finds the evidence, while the LLM explains the evidence. This makes the answers grounded in the provided knowledge instead of relying only on the LLM's internal knowledge.
>**Retriever** = Librarian (Knows exactly which pages to bring you.)
>**LLM** = Teacher (Reads those pages and explains them in simple language.)

3. Why can't an LLM simply process a sentence one word at a time from left to right? Why is self-attention necessary?
>Self-attention is necessary because the meaning of a word depends on the other words in the sentence. If an LLM only processed words one at a time from left to right, it would struggle to understand long-range relationships and determine which earlier words are most relevant. Self-attention lets every word look at the other words in the context and assign different levels of importance to them. This allows the model to build a context-aware representation before predicting the next token.

4. What is a context window, and why does it create the need for chunking in RAG systems?
>A context window is the amount of information an LLM can consider while generating a response. It acts like the model's working memory and includes elements such as the system prompt, conversation history, user query, and retrieved documents.
>Chunking is needed because documents are often large or contain multiple topics. Splitting them into smaller, meaningful chunks allows the retriever to find only the most relevant information instead of sending an entire document to the LLM. This improves retrieval accuracy, reduces unnecessary context, and produces more grounded answers.

5. Why must the user's query and the documents be embedded using the same embedding model? What problems would occur if you embedded the documents with one model and the query with a different model? 
>The query and the documents must be embedded using the same embedding model because the model defines the vector space in which semantic similarity is measured. If different embedding models are used, the vectors are generated in different embedding spaces and are no longer directly comparable. As a result, similarity search becomes unreliable, causing relevant documents to be missed or irrelevant documents to be retrieved.

6. Why don't we generate one embedding for an entire document? Why do RAG systems split documents into smaller chunks before creating embeddings?
>We don't create one embedding for an entire document because a single vector would have to represent many different topics at once, reducing retrieval precision. By splitting the document into smaller, semantically meaningful chunks, each embedding captures a specific concept. When a user asks a question, the vector database can retrieve only the chunks that are most relevant instead of the entire document. This reduces unnecessary context, improves retrieval accuracy, lowers token usage, and helps the LLM generate more focused and grounded answers.

7. What does a vector database actually store, and how does it determine which chunks are relevant to a user's question?
>A vector database stores the embedding vectors for document chunks along with the associated chunk text and optional metadata such as document IDs or source information. When a user asks a question, the query is converted into an embedding using the same embedding model. The vector database compares the query embedding with the stored embeddings using a similarity metric, ranks the chunks by semantic similarity, and returns the top matching chunks. It retrieves the most semantically relevant information but does not reason about or verify the correctness of that information.

8. Why do we include the retrieved chunks inside the prompt instead of simply asking the LLM the user's question directly? What problem does this solve?
>Retrieved chunks are included in the prompt to provide the LLM with relevant, up-to-date evidence before it generates an answer. Without this context, the LLM must rely only on its pre-trained knowledge, which may be incomplete, outdated, or incorrect for the user's specific documents. By grounding the prompt with retrieved chunks and instructing the model to answer based only on that context, RAG produces more accurate, verifiable, and less hallucinatory responses.
